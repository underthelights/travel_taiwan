# LLM Wiki - Taiwan Travel Planner

이 파일은 단순 프로젝트 설명서가 아니라, 앞으로 매일 여행 정보가 하나씩 추가될 때 LLM 에이전트가 같은 방식으로 읽고, 판단하고, 기록하기 위한 위키 인덱스다.

`agents.md`는 운영 규칙과 지식 구조를 담당한다. 실제 여행 내용은 `data.js`와 `CLAUDE.md`에 쌓는다.

## 1. 먼저 읽을 것

새 세션에서 작업을 시작하면 아래 순서로 읽는다.

1. `agents.md` - LLM 작업 규칙과 위키 구조.
2. `CLAUDE.md` - 현재 확정 상태, 결정 큐, 진행 메모.
3. `data.js` - 실제 사이트가 렌더링하는 단일 데이터 소스.
4. 관련 HTML/CSS/JS - 화면이나 렌더링 변경이 필요할 때만.

`CLAUDE.md`와 `data.js`가 충돌하면, 사이트에 반영되는 기준은 `data.js`다. 다만 사용자가 확정한 최신 말이 있으면 그 말을 우선하고 두 파일을 맞춘다.

## 2. 문서 역할

| 파일 | 역할 |
| --- | --- |
| `agents.md` | LLM 위키 운영 규칙, 기록 방식, 저장 위치 안내. 메타 규칙만 쓴다. |
| `CLAUDE.md` | 현재 여행 상태, 확정 내역, 미정 항목, 조사 메모, 매일 누적되는 운영 로그. |
| `data.js` | 사이트 렌더링 단일 소스. 일정, 장소, 결정 큐, 음식, 교통, 예산, 팁. |
| `README.md` | 공개용 짧은 프로젝트 소개. 상세 운영 지식은 넣지 않는다. |
| `taiwan_trip.kml` | `data.js`에서 생성된 Google My Maps import 파일. 수동 편집하지 않는다. |

## 3. 현재 사이트 구조

| 파일 | 역할 |
| --- | --- |
| `index.html` | 홈. 히어로, 진행률, 항공/예산, 일차 카드, 전체 Leaflet 지도. |
| `day.html` | `day.html?d=1/2/3` 일차별 상세. 타임라인, 지도, 상세 패널, 분류 필터, 구글맵 경로, KML 내보내기. |
| `taiwan-notes.html` | 대만 배경지식, 도시 역할, 현지 규칙을 정리한 핵심노트. |
| `stay.html` | `TRIP.stayInfo` 기반 타이베이 숙소 권역, 후보 호텔, 예약 체크 기준. |
| `eat.html` | `TRIP.eats` 기반 먹킷리스트. |
| `train.html` | `TRIP.trainInfo` 기반 교통카드, 패스, 고속철 구간권, 이동 방식 가이드. |
| `info.html` | 준비물 전용 페이지. 서류, 수하물, 현금/통신, 지도 저장, 예산만 다룬다. |
| `nav.js` | 공통 내비게이션 보강. 대만 핵심노트 링크 삽입, 일차별 분류 팝오버 생성. |
| `style.css` | 공통 스타일. 다크/게임풍 여행 플래너 톤 유지. |
| `build_kml.js` | `data.js`를 읽어 `taiwan_trip.kml` 재생성. |
| `assets/` | favicon, OG 이미지, 홈 히어로 등 래스터 에셋. |

## 4. 지식 저장 원칙

매일 들어오는 새 정보는 아래 기준으로 저장한다.

| 정보 종류 | 저장 위치 | 처리 |
| --- | --- | --- |
| 사용자가 확정한 항공, 숙소, 교통, 예약 | `data.js` + `CLAUDE.md` | `decisions`를 `done`으로 바꾸고 관련 장소를 `confirmed`로 올린다. |
| 아직 후보인 장소, 맛집, 일정 아이디어 | `data.js`의 후보 데이터 또는 `CLAUDE.md` 조사 메모 | 확정 처리하지 않는다. 기본 상태는 `candidate`. |
| 교통카드, 패스, 고속철 예매 판단 | `data.js`의 `TRIP.trainInfo` + `CLAUDE.md` | 화면에 보여야 하면 `trainInfo`, 근거와 조사일은 `CLAUDE.md`. |
| 매일 조사한 링크, 근거, 비교 내용 | `CLAUDE.md` | 날짜와 주제별로 짧게 누적한다. |
| 사이트에 보여야 하는 일정/예산/팁 | `data.js` | HTML에 하드코딩하지 않는다. |
| 화면 구성, 스타일, UI 동작 | HTML/CSS/JS | 데이터와 렌더링 책임을 섞지 않는다. |
| LLM 작업 방식 자체의 변경 | `agents.md` | 매일 여행 내용은 넣지 않는다. |

## 5. 매일 업데이트 루틴

사용자가 "오늘 이거 추가해", "이거 후보로 넣어", "이걸로 확정"처럼 말하면 다음 순서로 처리한다.

1. 정보 성격을 분류한다: `확정`, `후보`, `조사 메모`, `화면 기능`, `에셋`.
2. 관련 기존 항목을 먼저 찾는다: `rg "키워드" data.js CLAUDE.md *.html`.
3. 중복이 있으면 기존 항목을 갱신하고, 없으면 새 항목을 추가한다.
4. 확정이 아닌 정보는 `confirmedPlaceIds`에 넣지 않는다.
5. 장소, 좌표, 일차, 동선 순서가 바뀌면 `node build_kml.js`를 실행한다.
6. 확정 상태나 결정 큐가 바뀌면 `CLAUDE.md`의 현재 상태와 진행 메모도 갱신한다.
7. 마지막에 변경 파일과 검증 결과를 사용자에게 짧게 보고한다.

## 6. 확정 처리 규칙

사용자가 명시적으로 확정한 경우에만 아래를 수행한다.

1. `TRIP.decisions[]`의 관련 항목을 `status: 'done'`으로 변경.
2. `note`에 확정 내용을 구체적으로 기록.
3. 관련 장소를 추가/수정하고, 확정된 장소 ID를 `confirmedPlaceIds`에 추가.
4. 장소/좌표/동선 변경이 있으면 `node build_kml.js` 실행.
5. `CLAUDE.md`의 확정 상태와 TODO 체크를 갱신.

금지: "좋아 보임", "추천", "후보" 수준의 정보를 확정처럼 처리하지 않는다.

## 7. 데이터 모델 메모

`data.js`는 브라우저와 Node 양쪽에서 동작해야 한다. `build_kml.js`가 fake `window`로 `data.js`를 `eval`하기 때문이다.

장소 기본 필드:

```js
{
  id: 19,
  day: 1,
  kind: 'food',
  time: '18:00',
  name: '한글 장소명',
  nameEn: 'English Name',
  addr: '주소',
  cat: '표시용 분류',
  stay: '1시간',
  lat: 25.0000,
  lng: 121.0000,
  move: {mode: '🚇 MRT', min: '20분', cost: 'NT$25', note: '이동 설명'},
  desc: '장소 설명',
  tips: '팁'
}
```

현재 유효한 `kind` 값:

| kind | 의미 |
| --- | --- |
| `flight` | 항공, 공항 입출국 |
| `move` | 고속철, MRT, 버스, 택시 등 이동 |
| `food` | 식사, 야시장, 디저트 |
| `stay` | 숙소, 체크인, 숙박 |
| `sight` | 관광지, 사원, 전망대, 산책 |
| `shop` | 쇼핑, 기념품 |

상태는 장소 객체 안에 직접 쓰지 않는다. `confirmedPlaceIds`, `tbdPlaceIds`, `window.placeStatus(id)`로 판단한다.

## 8. CLAUDE.md 기록 방식

매일 쌓이는 여행 위키성 내용은 `CLAUDE.md`에 넣는다. 필요하면 아래 섹션을 유지하거나 추가한다.

```md
## Daily Log

### 2026-06-27
- 조사:
- 후보:
- 확정:
- 다음 결정:
```

조사 메모는 길게 붙여넣지 말고, 나중에 의사결정에 필요한 수준으로 요약한다.

권장 형식:

```md
- [후보] 시먼딩 A호텔: 위치 좋음, 가격대 OOO, 단점 OOO. 숙소 결정 전 비교 필요.
- [확정] 항공권: LJ0735/LJ0738, 2026.08.01~08.03, ₩298,700.
- [주의] 귀국날 17:30 비행 기준 15:00 RMQ 도착 목표.
```

## 9. 검증 명령

빌드 시스템은 없다.

```bash
node build_kml.js
```

장소/좌표/동선이 바뀐 경우 KML을 재생성한다.

```bash
python3 -m http.server 8000
```

정적 페이지 로컬 확인용 서버다. 필요할 때만 실행한다.

확인 대상:

- `index.html`: 홈 지도, 진행률, 항공/예산, 일차 카드.
- `day.html?d=1`, `day.html?d=2`, `day.html?d=3`: 지도, 타임라인, 상세 패널, 분류 필터.
- `eat.html`, `train.html`, `info.html`: `data.js`에서 렌더링되는 목록.
- `taiwan-notes.html`: 정적 핵심노트 레이아웃.

## 10. 작업 안전 규칙

- 작업트리에 사용자 변경이 있을 수 있다. 관련 없는 변경은 되돌리지 않는다.
- 큰 구조 변경보다, 매일 누적 가능한 작은 변경을 선호한다.
- HTML에 일정/예산/장소 데이터를 새로 하드코딩하지 않는다.
- `taiwan_trip.kml`은 가능하면 `node build_kml.js`로만 갱신한다.
- 여행 정보가 최신성에 민감하면 확인 날짜와 출처를 `CLAUDE.md` 조사 메모에 남긴다.
- 사용자가 한국어로 관리하므로 UI 문구와 위키 메모는 한국어를 기본으로 한다.
